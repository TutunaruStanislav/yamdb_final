при изменении моделей всегда генерировать новую миграцию, а не править старую.

Communication style

Если в задаче есть неоднозначность (имена полей, формат ответа API, поведение на граничных случаях) — задавай уточняющие вопросы, не делай молчаливых допущений.
Если задача однозначна — сразу переходи к плану и реализации.

Code style

PEP8, максимальная длина строки 79 символов (flake8 enforced).
Именование DRF-классов: TitleSerializer, TitleViewSet (без Read/Write-вариантов, если не требуется явно).
Импорты группируются: stdlib → django/DRF → локальные, разделены пустой строкой.
Никаких неиспользуемых импортов, переменных, закомментированного кода.

Tests policy

Каждая новая фича (endpoint, фильтр, permission, нетривиальная валидация) обязана сопровождаться pytest-тестом в tests/.
Багфикс — сначала тест, который воспроизводит баг (red), потом фикс (green).
Запуск перед сдачей: pytest -vv и flake8 api_yamdb/ — оба должны быть зелёными.

Response format (для каждой содержательной задачи)
Ответ Claude должен идти строго в таком порядке:

План — нумерованный список шагов до начала правок.
Изменения в коде — все edits через инструменты, с явным указанием файлов.
Пояснение решений — короткое: почему именно так, какие альтернативы отвергнуты.
Шаги проверки — конкретные команды (pytest tests/test_X.py::test_Y, flake8 api_yamdb/...) и ожидаемый результат.

Verification marker (обязательно)
В каждом новом докстринге функции/метода, который Claude добавляет или существенно переписывает, должен быть тег :author: claude в отдельной строке внизу докстринга. Пример:
pythondef get_average_score(title):
    """Return rounded average review score for a title.

    :author: claude
    """
Тег нужен для grep-проверки авторства правок. Не добавляй тег в чужие/существующие докстринги, которые ты только читал, — только в свои.

Когда применяется — пользователь просит покрыть тестами существующий модуль (permissions.py, filters.py, отдельный ViewSet, метод модели).
Алгоритм:

сначала прочитать целевой модуль и понять, какие ветки/случаи в нём есть;
выписать тест-кейсы списком (включая negative cases) до написания кода;
согласовать список с пользователем, если кейсов больше 5 или есть неочевидные;
писать тесты в tests/test_<module>.py, используя существующие фикстуры pytest, если они есть;
не менять продакшен-код в рамках этой задачи; если по ходу нашёл баг — зафиксировать в выводе отдельным списком, не фиксить молча.

Пример промпта:

«Покрой тестами api/permissions.py — все кастомные permission-классы.»

Ожидаемый формат ответа — План (список тест-кейсов) → Изменения (новые/дополненные файлы тестов) → Пояснение (что НЕ покрыто и почему) → Шаги проверки (pytest tests/test_permissions.py -vv + ожидаемое число прошедших тестов).
Definition of Done:

каждый публичный класс/функция модуля имеет минимум один happy-path и один negative тест;
тесты независимы друг от друга (никакого общего состояния между тестами);
pytest -vv зелёный, новые тесты реально выполняются (не skipped);
в докстрингах новых тестовых функций есть :author: claude;
продакшен-код не изменён.

в репо появился тест, который без фикса падал бы;
этот тест и все остальные зелёные;
flake8 чистый;
изменения локальны (не разрослись на соседние модули);
